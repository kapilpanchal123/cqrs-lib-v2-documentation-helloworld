Mathematical Formalism

Formal Command Pipeline Model

CQRS models command execution as a functional pipeline \(\mathcal{P}\), where each stage \(s \in S\) is orchestrated through dynamically extensible interceptor chains. The following definitions formalize the framework's behavior.

Core Domain Sets

Domain Definitions
\[ \begin{aligned} C &= \text{set of all Commands} \\ R &= \text{set of all Results} \\ E &= \text{set of all Exceptions} \\ H &= \text{set of all CommandHandlers} \\ \mathcal{S} &= \{s : s: \varnothing \to R \cup E\}\ (\text{Suppliers}) \end{aligned} \]

Core Pipeline Mappings

Transformation Functions
\[ \begin{aligned} \text{route} &: C \to H \\ \text{exec} &: C \to \mathcal{S} \\ \text{pre}_{m} &: C \to C \\ \text{post}_{p} &: C \times (R \cup \{\bot\}) \to R \cup \{\bot\} \end{aligned} \]

Problem Statement

Traditional Tight-Coupling (Anti-Pattern)
\[ s_i = B(s_i) + O(s_i), \quad \forall i = 1, \dots, n \]

Where \(B(s_i)\) is business logic and \(O(s_i)\) are operational concerns — dangerously entangled.

EOIW Separation (Solution)
\[ s_i = B(s_i), \quad I_i = O(s_i), \quad \forall i \]

Interceptor chain \(I_i\) handles operational concerns independently.

Composed Pipeline Function

Pipeline Composition
\[ \mathrm{pipeline} = \mathrm{wrapPost} \circ \mathrm{exec} \] \[ \mathrm{wrapPost}(s)() = \mathrm{finalize}\Big( \text{fold}_{p \in P}\big(\mathrm{post}_p,\, \mathrm{tryExec}(s)\big) \Big) \]

Execution Semantics

Handler Invocation Outcome
\[ \mathrm{tryExec}(s) = \begin{cases} (s(), \bot) & \text{if handler succeeds}\\[6pt] (\bot, e) & \text{if handler throws } e \end{cases} \]

Temporal Property

Deterministic Ordering Guarantee
\[ \forall c \in C,\, \exists! r \in R, e \in E: \] \[ \underbrace{\mathrm{pre}_{m_1} \circ \cdots \circ \mathrm{pre}_{m_n}}_{\text{middleware orchestration}}(c) \;\to\; \underbrace{\mathrm{exec}_h(c)}_{\text{handler}} \;\to\; \underbrace{\mathrm{post}_{p_1} \circ \cdots \circ \mathrm{post}_{p_m}}_{\text{post-processing}}(c,r,e) \]

Resilience Extension

Failure + Retry Semantics
\[ \forall c \in C,\ \exists! p \in P : \Big(\mathrm{exec}(c) \prec \mathrm{post}_p(c,r,e)\Big) \ \lor\ \Big(\mathrm{fail}(c) \Rightarrow \Diamond\, \mathrm{retry}(c)\Big) \]
Execution Model

Pipeline Lifecycle & Transaction Strategy

Each command traverses a structured five-phase lifecycle. Savepoints ensure business logic rollbacks never erase audit records, while independent transactions guarantee failure states persist even when outer transactions abort.

1
Middleware Execution
Command status is set to PROCESSING. All registered CommandMiddleware components are invoked in priority order. The default DefaultCommandPersistenceMiddleware saves the command record to the database.
2
Handler Resolution
DefaultCommandRouter scans registered handlers via O(n) linear scan, calling matches(command) on each. The first matching handler is returned as a deferred Supplier<RES>.
3
Savepoint Creation
A JDBC savepoint "preHandlerSavepoint" is created. This checkpoint preserves all prior writes (including the PROCESSING record) and allows business logic to be rolled back independently.
4
Handler + Post-Processing
handler.handle(command) executes the business logic. On success, all CommandPostProcessor instances run sequentially. The default post-processor calls update(id, COMPLETED).
5
Commit & Result Delivery
Savepoint is released. Transaction commits. The PipelineResult.success(RES) value is extracted via getOrThrow() and returned to the caller.

Failure Handling Strategy

⚡
Execution Failure (middleware/routing): Handled in an independent transaction via executeIndependent(). This guarantees the FAILED state is persisted even if the outer transaction rolls back entirely.
↩
Handler Failure: Rolls back to preHandlerSavepoint. Domain writes are undone. Failure strategies run within the still-open transaction and update status to FAILED. Transaction commits with FAILED state. Exception re-thrown after commit.
↩
Post-Processing Failure: Same savepoint rollback path. CommandFailureStage.POST_PROCESSING strategies execute. Core business logic result is already committed prior to savepoint — only post-processor writes are lost.
💡
Key Guarantee: PipelineResult.failure() is returned (not thrown) inside the transaction block. This forces the transaction to commit with FAILED status before the exception is re-thrown via getOrThrow().

Core Abstractions

Component Reference

Every component is designed around a single, well-defined responsibility. Override or compose these interfaces to customize the framework's behavior.

interface
CH
CommandHandler<REQ, RES>
Core processing unit. Receives a Command<REQ>, executes domain logic, returns RES. The matches() default uses TypeToken raw-type assignability.
abstract class
AC
AbstractCommandHandler<REQ, RES>
Captures generic type parameters at runtime via Guava TypeToken, solving Java's type-erasure. Extend this instead of implementing the interface directly.
interface
CP
CommandPipeline
Entry point. send(command) returns a lazy Supplier<RES>. Execution is deferred until .get() is called, enabling composition with transactions and async wrappers.
interface
CM
CommandMiddleware
Pre-execution hook. Invoked sequentially before handler resolution. Used for logging, validation, persistence, metadata enrichment, and security. Sorted via Ordered interface.
interface
PP
CommandPostProcessor<T>
Post-execution hook. Runs after successful handler completion. Functional interface — supports lambdas. Default implementation updates status to COMPLETED.
interface
FS
CommandFailureStrategy
supports(stage) filters strategies by CommandFailureStage. onFailure(cmd, t) reacts. Multiple strategies per pipeline. Default implementations update status to FAILED.
interface
CS
CommandStore
Persistence abstraction. save(cmd, status) creates records, update(id, status) and update(id, status, throwable) track lifecycle transitions.
interface
TX
CommandTransactionManager
execute() joins existing or starts new transactions. executeIndependent() always creates a fresh transaction — critical for durable failure state writes.
sealed interface
PR
PipelineResult<T>
Sealed type with Success<T> and Failure<T> records. getOrThrow() extracts value or re-throws. Enables transaction-commit-before-throw pattern.
Integration Guide

Using the Framework

A complete walkthrough of integrating CQRS into a Spring Boot application — from defining command types to wiring the pipeline and handling HTTP requests.

Step 1 — Create the Database Schema - (Resources Directory)

First, create your database schema, make sure that the application is able to communicate with the database. The example command schema is attached inside the resources directory.

sql · cqrs_master.sql 
              CREATE TABLE cqrs_master (
              id              UUID PRIMARY KEY,
              idempotency_key VARCHAR(64),
              status          VARCHAR(255),
              tenant_id       VARCHAR(64),
              username        VARCHAR(255),
              request_url     TEXT,
              class_name      VARCHAR(255),
              api_version     VARCHAR(32),
              correlation_id  VARCHAR(255),
              error           TEXT,
              payload         TEXT,
              created_at      TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
              updated_at      TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,

              CONSTRAINT chk_idempotency_key_correlation_id
                  UNIQUE(idempotency_key, correlation_id),

              CONSTRAINT chk_status
                  CHECK (
                      status IN ('INIT', 'PENDING', 'PROCESSING', 'COMPLETED', 'FAILED')
                  )
          );
💡
Keep your Command subclass thin — it is a carrier, not a handler. All business logic belongs in the handler. The command's metadata fields (correlationId, tenantId, etc.) are inherited from Command<T>.

Step 3 — Springboot Configuration for the creation of Spring bean

java · CqrsConfigModuleV1.java 
        @Configuration
        public class CqrsConfigModuleV1 {

          @Bean
          @ConditionalOnMissingBean
          public ObjectMapper objectMapper() {
            return new ObjectMapper().findAndRegisterModules();
          }

          @Bean
          @ConditionalOnMissingBean
          public CommandJsonMapper commandJsonMapper(
              final ObjectMapper objectMapper) {
            return new CommandJsonMapper(objectMapper);
          }

          @Bean
          @ConditionalOnMissingBean
          public CommandTransactionManager commandTransactionManager(
              final DataSource dataSource) {
            return new JdbcCommandTransactionManager(dataSource);
          }

          @Bean
          @ConditionalOnMissingBean
          public CommandStore commandStore(
              final CommandJsonMapper commandJsonMapper,
              final DataSource dataSource) {
            return new DefaultCommandService(commandJsonMapper, dataSource);
          }

          @Bean
          @ConditionalOnMissingBean
          public CommandModule commandModule(
              final List<CommandHandler<?,?>> handlers,
              final List<CommandMiddleware> middlewares,
              final List<CommandPostProcessor<?>> postProcessors,
              final List<CommandFailureStrategy> failureStrategies,
              final CommandStore commandStore,
              final CommandTransactionManager commandTransactionManager) {

            return CommandModuleBuilder.create()
                .handlers(handlers)
                .middlewares(middlewares)
                .postProcessors(postProcessors)
                .failureStrategies(failureStrategies)
                .commandStore(commandStore)
                .transactionManager(commandTransactionManager)
                .build();
          }

          @Bean
          @ConditionalOnMissingBean
          public CommandPipeline commandPipeline(
              final CommandModule commandModule) {
            return commandModule.getPipeline();
          }
        }

Step 6 — Define Your Command Handler

Note: Do not use the @Transactional from Spring boot as the transactions in the framework are programmetically invoked inside the DefaultCommandPipeline object.

java · UserCommandHandler.java 
package org.example.handler;

        @RequiredArgsConstructor @Component
        public class UserCommandHandler 
            extends AbstractCommandHandler<UserMasterRequest, UserMasterResponse> {

            private final UserMasterService service;

            @Override
            public UserMasterResponse handle(
                final Command<UserMasterRequest> command) {

                final Map<String, String> responseMap = service.savePayload(command);

                return UserMasterResponse.builder()
                    .response(responseMap)
                    .build();
            }
        }

Step 7 — Extend Your Command PostProcessors with Resilience(Optional)

If you do define your own post processor make sure it is updated inside the spring configuration file.

java · ResiliencePostProcessor.java 
package org.example.postprocessor;

          @Slf4j @RequiredArgsConstructor
          public class ResiliencePostProcessor<T> {

          private final CommandStore commandStore;
          private final CommandTransactionManager transactionManager;
          private final AtomicInteger counter = new AtomicInteger(0);

          @Retry(name = "run", fallbackMethod = "updateCommandToFailedOnRetryExhaustion")
          public void run(final Command<T> command) {

          int attempt = counter.incrementAndGet();
          log.info("Attempt {} for command {}", attempt, command.getId());

          // Simulates a Failed Condition comment for real code
          if (attempt < 3) {
              throw new RuntimeException("Simulated transient failure");
          }

          commandStore.update(command.getId(), CommandStatus.COMPLETED);
          log.info("Command {} marked as COMPLETED", command.getId());
            }

                public void updateCommandToFailedOnRetryExhaustion(
                final Command<T> command, 
                final Throwable t) {

                log.error("Retries exhausted for command {}. Marking as FAILED. Reason: {}",
                    command.getId(), t.getMessage());
                    commandStore.update(command.getId(), CommandStatus.FAILED, t);

                throw new RuntimeException(t);
            }
          }

Step 8 — Usage inside the API

java · CommandController.java 
        @RequiredArgsConstructor
        @RestController
        @RequestMapping(path = "/api")
        public class CommandController {

          private final CommandPipeline pipeline;

          @PostMapping(path = "/{version}/add-user-master", version = "2.0")
          public ResponseEntity<UserMasterResponse> addUserMaster(
              final @RequestHeader(value = CommandConstants.COMMAND_IDEMPOTENCY_HEADER, required = true) UUID idempotencyKey,
              final @PathVariable("version") String version,
              final HttpServletRequest httpRequest,
              final @RequestBody UserMasterRequest request) {

            final String correlationId =
                (String) httpRequest.getAttribute(CommandConstants.COMMAND_CORRELATION_ID_HEADER);

            final UserCommand command = new UserCommand();
            command.setId(UUID.randomUUID());
            command.setIdempotencyKey(idempotencyKey.toString());
            command.setCorrelationId(correlationId);
            command.setRequestUrl(httpRequest.getRequestURL().toString());
            command.setApiVersion(version + ".0");
            command.setClassName(UserMasterRequest.class.getName());
            command.setPayload(request);
            command.setCreatedAt(OffsetDateTime.now());

            final Supplier<UserMasterResponse> response =
                pipeline.send(command);

            final UserMasterResponse userMasterResponse = response.get();

            return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(userMasterResponse);
          }
        }

Interface & Class Quick Reference

Type Kind Package Key Methods Notes
Command<T> class core getters/setters for all fields Serializable. Not thread-safe. One per request.
CommandHandler<REQ,RES> interface core handle(cmd), matches(cmd), requestTypeToken() Extend AbstractCommandHandler instead.
AbstractCommandHandler<REQ,RES> abstract core requestTypeToken(), responseTypeToken() Uses Guava TypeToken to capture generic params.
CommandPipeline interface core send(command): Supplier<RES> Entry point. Returns lazy Supplier.
CommandMiddleware interface core invoke(command): void Pre-handler. Implement Ordered for priority.
CommandPostProcessor<T> @FunctionalInterface core run(command): void Post-handler. Runs after successful execution.
CommandStore interface core save, update(id,status), update(id,status,t) Persistence abstraction. Implement for custom storage.
CommandTransactionManager interface transactions execute(action), executeIndependent(action) Decouple from JDBC. Implement for JPA/Spring TX.
CommandSavepointManager interface transactions savePoint, rollbackToSavepoint, releaseSavepoint Partial rollback within active transaction.
CommandFailureStrategy interface failure supports(stage), onFailure(cmd, t) Pluggable. Multiple per pipeline. Stage-filtered.
PipelineResult<T> sealed core success(v), failure(ex), getOrThrow() Commit-before-throw pattern carrier.
CommandStatus enum core INIT, PENDING, PROCESSING, COMPLETED, FAILED Immutable. Thread-safe.
CommandModuleBuilder final class infrastructure create(), handler(), build() Fluent builder. Not thread-safe. Init-time only.
JdbcCommandTransactionManager final class jdbc execute(), executeIndependent() Thread-safe. Uses JdbcConnectionHolder (ThreadLocal).
Ordered interface utils getOrder(): int Lower = higher priority. Implement alongside Middleware / PP.

Default Implementations Auto-Applied by CommandModuleBuilder

middleware
DefaultCommandPersistenceMiddleware
Saves command with PROCESSING status before execution. Applied when CommandStore is configured and no custom middleware provided.
post-processor
DefaultCommandPostProcessor
Updates command status to COMPLETED after successful execution. Applied when CommandStore is configured and no custom post-processor provided.
failure
DefaultCommandExecution/PostProcessorFailure
Updates command status to FAILED with full stack trace for both failure stages. Applied when CommandStore configured and no custom strategy provided.
