Mathematical Formalism

Formal Command Pipeline Model

EOIW models command execution as a functional pipeline \(\mathcal{P}\), where each stage \(s \in S\) is orchestrated through dynamically extensible interceptor chains. The following definitions formalize the framework's behavior.

Core Domain Sets

Domain Definitions
\[\begin{aligned}C &= \text{set of all Commands}\\R &= \text{set of all Results}\\E &= \text{set of all Exceptions}\\H &= \text{set of all CommandHandlers}\\\mathcal{S} &= \{s : s: \varnothing \to R \cup E\}\quad(\text{Suppliers})\end{aligned}\]

Core Pipeline Mappings

Transformation Functions
\[\begin{aligned}\text{route} &: C \to H\\\text{exec} &: C \to \mathcal{S}\\\text{pre}_{m} &: C \to C\\\text{post}_{p} &: C \times (R \cup \{\bot\}) \to R \cup \{\bot\}\end{aligned}\]

Problem Statement

Traditional Tight-Coupling (Anti-Pattern)
\[s_i = B(s_i) + O(s_i), \quad \forall i = 1, \dots, n\]

Where \(B(s_i)\) is business logic and \(O(s_i)\) are operational concerns — dangerously entangled.

EOIW Separation (Solution)
\[s_i = B(s_i), \quad I_i = O(s_i), \quad \forall i\]

Interceptor chain \(I_i\) handles operational concerns independently.

Composed Pipeline Function

Pipeline Composition
\[\mathrm{pipeline} = \mathrm{wrapPost} \circ \mathrm{exec}\] \[\mathrm{wrapPost}(s)() = \mathrm{finalize}\Big(\text{fold}_{p \in P}\big(\mathrm{post}_p,\,\mathrm{tryExec}(s)\big)\Big)\]

Execution Semantics

Handler Invocation Outcome
\[\mathrm{tryExec}(s)=\begin{cases}(s(),\bot)&\text{if handler succeeds}\\[6pt](\bot,e)&\text{if handler throws }e\end{cases}\]

Temporal Property

Deterministic Ordering Guarantee
\[\forall c \in C,\,\exists! r \in R, e \in E:\] \[\underbrace{\mathrm{pre}_{m_1}\circ\cdots\circ\mathrm{pre}_{m_n}}_{\text{middleware orchestration}}(c)\;\to\;\underbrace{\mathrm{exec}_h(c)}_{\text{handler}}\;\to\;\underbrace{\mathrm{post}_{p_1}\circ\cdots\circ\mathrm{post}_{p_m}}_{\text{post-processing}}(c,r,e)\]

Resilience Extension

Failure + Retry Semantics
\[\forall c \in C,\ \exists! p \in P:\Big(\mathrm{exec}(c)\prec\mathrm{post}_p(c,r,e)\Big)\ \lor\ \Big(\mathrm{fail}(c)\Rightarrow\Diamond\,\mathrm{retry}(c)\Big)\]
Execution Model

Pipeline Lifecycle & Transaction Strategy

Each command traverses a structured five-phase lifecycle. Savepoints ensure business logic rollbacks never erase audit records. Independent transactions guarantee failure states persist even when outer transactions abort.

1
Middleware Execution
Command status is set to PROCESSING by DefaultSynchronousCommandExecutor. All registered CommandMiddleware instances are invoked sequentially in Ordered priority. The default DefaultCommandPersistenceMiddleware calls save(cmd, PROCESSING) .
2
Handler Resolution
DefaultCommandRouter scans handlers via O(n) linear scan, calling matches(command) on each. Uses requestTypeToken().getRawType().isAssignableFrom() for type matching. The first match is returned as a deferred Supplier<RES>.
3
Savepoint Creation
A JDBC savepoint "preHandlerSavepoint" is created by the JdbcCommandSavepointManager . This checkpoint preserves the PROCESSING record while isolating all subsequent business writes for potential rollback.
4
Handler + Post-Processing
handlerSupplier.get() invokes the business logic. On success, all CommandPostProcessor instances run sequentially. The default DefaultCommandPostProcessor calls update(id, COMPLETED). Both handler failures and post-processor failures are caught by the same block and use CommandFailureStage.POST_PROCESSING.
5
Result Delivery
Savepoint is released. PipelineResult.success(RES) is returned. Transaction commits. getOrThrow() extracts the value. On failure the transaction also commits — with FAILED status — before re-throwing via getOrThrow().

Failure Handling — Accurate Stage Mapping

⚡
EXECUTION Stage Failure (middleware or routing): Strategies that return true from supports(EXECUTION) are invoked via executeIndependent() — a brand-new transaction committed in isolation, ensuring the FAILED record persists even as the outer transaction rolls back entirely.
↩
POST_PROCESSING Stage Failure (handler or post-processor): Both failure types are caught by the same try-catch block in DefaultCommandPipeline and both use CommandFailureStage.POST_PROCESSING. Rolls back to preHandlerSavepoint, domain writes are undone, failure strategies run in the still-open transaction, then PipelineResult.failure() is returned (not thrown) so the transaction commits with FAILED status before the exception is re-thrown.
💡
Key Design: PipelineResult.failure(wrapped) is returned inside the transaction, not thrown. This forces the transaction to commit with FAILED status before getOrThrow() re-throws the exception outside the transaction boundary.
⚠️
NoOpCommandTransactionManager Limitation: executeIndependent() in NoOpCommandTransactionManager returns null without executing the action. This means EXECUTION-stage failure strategies are silently skipped when using the NoOp manager. Use JdbcCommandTransactionManager in production.

Core Abstractions

Component Reference

Every component is designed around a single, well-defined responsibility. Override or compose these interfaces to customize the framework's behavior without touching the pipeline core.

interface
CH
CommandHandler<REQ, RES>
Core processing unit. Receives Command<REQ>, executes domain logic, returns RES. Default matches() uses requestTypeToken().getRawType().isAssig... — raw type check only.
abstract class
AC
AbstractCommandHandler<REQ, RES>
Captures REQ and RES generic types at construction time via Guava TypeToken, solving Java's type-erasure. Extend this rather than implementing CommandHandler directly.
interface
CP
CommandPipeline
send(command) returns a lazy Supplier<RES>. Execution is deferred until .get() is called, enabling composition with async wrappers, retries, or monitoring at the call site.
interface
CM
CommandMiddleware
Pre-execution hook invoked sequentially before handler resolution. Used for logging, validation, persistence, metadata enrichment. Implement Ordered to control position in the chain.
@FunctionalInterface
PP
CommandPostProcessor<T>
Post-execution hook. Runs after successful handler completion. Failures here are caught by the same catch block as handler failures and use CommandFailureStage.POST_PROCESSING.
interface
FS
CommandFailureStrategy
supports(stage) filters strategies by CommandFailureStage. EXECUTION strategies run via executeIndependent(); POST_PROCESSING strategies run within the same open transaction.
interface
CS
CommandStore
Persistence abstraction. save(cmd, status) creates records. update(id, status) and update(id, status, throwable) track lifecycle transitions. DefaultCommandService manages its own JDBC transactions per operation.
interface
TX
CommandTransactionManager
execute() joins existing or starts new transactions via JdbcConnectionHolder ThreadLocal. executeIndependent() always creates a fresh connection — critical for durable failure state writes independent of outer tx outcome.
sealed interface
PR
PipelineResult<T>
Sealed type with Success<T> and Failure<T> records. getOrThrow() uses a switch expression. Enables the commit-before-throw pattern by returning failures instead of throwing within the tx boundary.
Integration Guide

Using the Framework

A complete walkthrough — from database schema to HTTP request handling in Spring Boot.

Step 1 — Create the Database Schema

The composite unique constraint on (idempotency_key, correlation_id) enforces idempotency at the database level.

sql · schema.sql 
CREATE TABLE public.cqrs_master (
  id              UUID PRIMARY KEY,
  idempotency_key VARCHAR(64),
  status          VARCHAR(255),
  tenant_id       VARCHAR(64),
  username        VARCHAR(255),
  request_url     TEXT,
  class_name      VARCHAR(255),
  api_version     VARCHAR(32),
  correlation_id  VARCHAR(255),
  error           TEXT,
  payload         TEXT,          -- JSON with @class metadata for polymorphic deserialization
  created_at      TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  updated_at      TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,

  CONSTRAINT chk_idempotency_key_correlation_id
      UNIQUE(idempotency_key, correlation_id),

  CONSTRAINT chk_status
      CHECK(status IN ('INIT', 'PENDING', 'PROCESSING', 'COMPLETED', 'FAILED'))
);
💡
The payload column stores JSON produced by CommandJsonMapper, which embeds a @class field containing the fully qualified class name for polymorphic reconstruction.

Step 2 — Correlation ID Filter

Generates and attaches a correlationId to every request as a request attribute. Extends OncePerRequestFilter to run exactly once per request.

java · CommandCorrelationFilter.java 
@Slf4j @Component
public class CommandCorrelationFilter extends OncePerRequestFilter {

  @Override
  protected void doFilterInternal(
      final HttpServletRequest request,
      final HttpServletResponse response,
      final FilterChain filterChain) throws ServletException, IOException {

    request.setAttribute(
        CommandConstants.COMMAND_CORRELATION_ID_HEADER,
        UUID.randomUUID().toString()
    );
    log.info("Correlation id: {}",
        request.getAttribute(CommandConstants.COMMAND_CORRELATION_ID_HEADER));
    log.info("Idempotency id: {}",
        request.getHeader(CommandConstants.COMMAND_IDEMPOTENCY_HEADER));

    filterChain.doFilter(request, response);
  }
}

Step 3 — Spring Boot Configuration

All beans use @ConditionalOnMissingBean — override any bean in your own @Configuration to customize behaviour without modifying the framework.

java · CqrsConfigModuleV1.java 
@Configuration
public class CqrsConfigModuleV1 {

  @Bean @ConditionalOnMissingBean
  public ObjectMapper objectMapper() {
    return new ObjectMapper().findAndRegisterModules();
  }

  @Bean @ConditionalOnMissingBean
  public CommandJsonMapper commandJsonMapper(final ObjectMapper objectMapper) {
    return new CommandJsonMapper(objectMapper);
  }

  @Bean @ConditionalOnMissingBean
  public CommandTransactionManager commandTransactionManager(final DataSource ds) {
    return new JdbcCommandTransactionManager(ds);
  }

  @Bean @ConditionalOnMissingBean
  public CommandStore commandStore(
      final CommandJsonMapper mapper, final DataSource ds) {
    return new DefaultCommandService(mapper, ds);
  }

  @Bean @ConditionalOnMissingBean
  public CommandModule commandModule(
      final List<CommandHandler<?,?>> handlers,
      final List<CommandMiddleware> middlewares,
      final List<CommandPostProcessor<?>> postProcessors,
      final List<CommandFailureStrategy> failureStrategies,
      final CommandStore commandStore,
      final CommandTransactionManager txManager) {

    return CommandModuleBuilder.create()
        .handlers(handlers)
        .middlewares(middlewares)         // empty list → DefaultCommandPersistenceMiddleware applied
        .postProcessors(postProcessors)   // empty list → DefaultCommandPostProcessor applied
        .failureStrategies(failureStrategies) // empty list → both defaults applied
        .commandStore(commandStore)
        .transactionManager(txManager)
        .build();
  }

  @Bean @ConditionalOnMissingBean
  public CommandPipeline commandPipeline(final CommandModule module) {
    return module.getPipeline();
  }
}

Step 4 — Request and Response Objects

⚠️
Do not use raw Java collection types (List<?>, Map<?,?> ) as your payload type. CommandJsonMapper embeds a @class field and uses Class.forName() for reconstruction — concrete, named classes are required.
java · UserMasterRequest.java + UserMasterResponse.java 
// Request DTO — your command's business payload
@Getter @Setter @NoArgsConstructor
public class UserMasterRequest {
  private String username;
  private String email;
  private String firstName;
  private String lastName;
  private int    age;
}

// Response DTO — returned after handler execution
@Getter @Setter @Builder @NoArgsConstructor @AllArgsConstructor
public class UserMasterResponse {
  private Map<String, String> response;
}

Step 5 — Extend Your Command Object

💡
Keep your Command subclass thin — it is a carrier, not a handler. All business logic belongs in the handler. Metadata fields (correlationId, tenantId, idempotencyKey, etc.) are inherited from Command<T>.

package org.example.command;

@Getter @Setter @NoArgsConstructor

Step 6 — Command Handler

🚫
Do not use Spring's @Transactional on handlers. Transactions are programmatically managed inside DefaultCommandPipeline via JdbcCommandTransactionManager . Mixing declarative and programmatic tx management will cause conflicts.
java · UserCommandHandler.java 
@RequiredArgsConstructor @Component
public class UserCommandHandler
    extends AbstractCommandHandler<UserMasterRequest, UserMasterResponse> {

  private final UserMasterService service;

  @Override
  public UserMasterResponse handle(final Command<UserMasterRequest> command) {
    final Map<String, String> result = service.savePayload(command);

    return UserMasterResponse.builder()
        .response(result)
        .build();
  }
}

Step 7 — Custom Post-Processor with Resilience4j (Optional)

Provide your own CommandPostProcessor bean to override the default. If supplied, DefaultCommandPostProcessor is not registered by the builder.

java · ResiliencePostProcessor.java 
@Slf4j @RequiredArgsConstructor
public class ResiliencePostProcessor<T> implements CommandPostProcessor<T> {

  private final CommandStore commandStore;
  private final AtomicInteger counter = new AtomicInteger(0);

  @Retry(name = "run", fallbackMethod = "onRetryExhaustion")
  @Override
  public void run(final Command<T> command) {
    int attempt = counter.incrementAndGet();
    log.info("Attempt {} for command {}", attempt, command.getId());

    // perform side-effect (e.g. event publishing)
    commandStore.update(command.getId(), CommandStatus.COMPLETED);
  }

  // Resilience4j fallback — exhausted retries
  public void onRetryExhaustion(final Command<T> command, final Throwable t) {
    log.error("Retries exhausted for command {}. Marking FAILED.", command.getId(), t);
    commandStore.update(command.getId(), CommandStatus.FAILED, t);
    throw new RuntimeException(t);
  }
}
💡
Because this post-processor failure is caught by DefaultCommandPipeline's Phase 3 catch block, the retry exhaustion exception will trigger the POST_PROCESSING failure strategies and roll back to the savepoint before marking the command FAILED.

Step 8 — REST Controller

The x-cqrs-idempotency-id header is required. The correlationId is read from the request attribute set by CommandCorrelationFilter.

java · CommandController.java 
@RequiredArgsConstructor @RestController @RequestMapping(path = "/api")
public class CommandController {

  private final CommandPipeline pipeline;

  @PostMapping(path = "/{version}/add-user-master")
  public ResponseEntity<UserMasterResponse> addUserMaster(
      @RequestHeader(value = CommandConstants.COMMAND_IDEMPOTENCY_HEADER, required = true) UUID idempotencyKey,
      @PathVariable("version") String version,
      HttpServletRequest httpRequest,
      @RequestBody UserMasterRequest request) {

    final String correlationId =
        (String) httpRequest.getAttribute(CommandConstants.COMMAND_CORRELATION_ID_HEADER);

    final UserCommand command = new UserCommand();
    command.setId(UUID.randomUUID());
    command.setIdempotencyKey(idempotencyKey.toString());
    command.setCorrelationId(correlationId);
    command.setRequestUrl(httpRequest.getRequestURL().toString());
    command.setApiVersion(version + ".0");
    command.setClassName(UserMasterRequest.class.getName());
    command.setPayload(request);
    command.setCreatedAt(OffsetDateTime.now());

    // pipeline.send() is lazy — execution begins on .get()
    final UserMasterResponse response = pipeline.<UserMasterRequest, UserMasterResponse>send(command).get();

    return ResponseEntity.status(HttpStatus.CREATED).body(response);
  }
}

middleware

post-processor
DefaultCommandPostProcessor
